Typical users:
-anonymous, ordinary authenticated users, admin users
-frequently different users = access to different sets of data (A-user shouldn’t read B-user’s emails)

User control focuses on the Big 3: 
-Authentication, session management, access control 
-each/together rep’s significant part of app’s attack surfaces, fundamental to appsec
-weakest link in the chain

Authentication
-basic dependency in app’s handling of user access: auth user = user is who he claims to be
-without auth: treat all users as anonymous/lower level of trust

basic auth:
-input un/pw, application checks for validity
-further auth models can be added as req’d: client certs, smart cards, etc.
-auth mech’s often support other fxns: self-registration, account recovery, pw change facility

Session Management
-after user logs in, next logical task is to manage auth’d user’s session as it makes a variety of HTTP requests from browser

Application needs: way to ID/process series of requests that originate form each unique user

Application does: 
  1. create session for each user
  2. issue user a token id’ing the session

What is a session: set of data structures held on server that track state of user’s interaction w/ app
-if user does not make requests w/in certain time, session expires

What is a token: unique string tied to user
-browser automatically submits token to server from user during each HTTP request
-token allows app to associate each request w/ each user

Ways of transmitting tokens:
-http cookies
-hidden form fields
-url query string

Alternatives to tokens:
-send user creds w/ each request
-store state info on client side rather than the server (encrypted)

Vulnerabilities in session management with tokens:
-defects in how tokens are generated, hacker can guess tokens issued
-defects in how tokens are handled, allowing token capture

Access Control
make/enforce correct decisions about whether user request should be permitted or denied

-if auth/session mgmt function correctly, access control can base access off that

-usually has detailed logic: specific to app area or functionality used, user role rules, 

Vulns in access control:
-devs make faulty assumptions with how users will interact with the app, omit access control for certain functions as a resultt
-this takes time (similar checks for each functionality), but is often effective
